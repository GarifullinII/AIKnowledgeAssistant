# AIKnowledgeAssistant

Проект-заготовка для AI-ассистента, который в дальнейшем будет работать как Telegram-бот и отвечать на вопросы по загруженным документам.

Основной сценарий:

1. Пользователь отправляет файл в Telegram.
2. Поддерживаемые форматы:
   - `PDF`
   - `Markdown`
   - `TXT`
3. Пользователь пишет вопрос, например:
   - `Что в этом документе?`
4. Бот анализирует содержимое документа и отвечает, например:
   - `В документе указано...`

## Цель проекта

Сделать систему, которая:

- принимает документ от пользователя;
- извлекает из него текст;
- передает текст в RAG-слой;
- отвечает на вопросы по содержимому документа;
- в дальнейшем интегрируется с Telegram-ботом.

На текущем этапе цель реализована частично: подготовлен backend-каркас, добавлены API для загрузки документа и получения списка документов, но реального анализа содержимого документа еще нет.
Приложение уже сохраняет загруженные файлы на диск, извлекает текст из `PDF`, `TXT` и `Markdown`, а также возвращает краткую информацию о сохраненном документе.

## Что есть на текущем этапе

В проекте уже сделано:

- создано приложение `FastAPI`;
- подключен роутер с префиксом `/api`;
- есть endpoint `GET /api/health`;
- есть endpoint `POST /api/ask`;
- есть endpoint `POST /api/upload`;
- есть endpoint `GET /api/documents`;
- логика ответа вынесена в отдельный сервис `rag_service`;
- логика загрузки документа вынесена в отдельный сервис `document_service`;
- добавлены Pydantic-схемы для запроса, ответа и документа;
- добавлен конфиг через `pydantic-settings`;
- добавлено простое in-memory хранилище документов;
- добавлена директория загрузки файлов `data/uploads`;
- добавлено извлечение текста из `PDF`, `TXT` и `MD`;
- в ответе по документу появились `stored_path`, `text_length` и `preview`.

Как это работает сейчас:

- `app/main.py` создает объект `FastAPI` и подключает роуты с общим префиксом `/api`;
- `app/api/routes.py` принимает HTTP-запросы и передает работу в сервисы;
- `app/schemas/query.py` описывает структуру запроса на `/api/ask` и структуру ответа;
- `app/schemas/document.py` описывает структуру ответа при загрузке документа и списка документов;
- `app/services/document_service.py` валидирует загружаемый файл, создает директорию загрузки, сохраняет файл на диск, извлекает текст, вычисляет размер, создает `id` документа и сохраняет метаданные;
- `app/db/memory_store.py` хранит список документов в памяти процесса;
- `app/services/rag_service.py` пока возвращает заглушку и не использует реальное содержимое документа;
- `app/core/config.py` теперь содержит настройку `upload_dir`, чтобы путь к каталогу загрузки был вынесен в конфиг.

Что реально происходит при вызове endpoint:

- `GET /api/health` возвращает `{"status": "ok"}`;
- `POST /api/upload` принимает файл, проверяет имя, тип или расширение, сохраняет его в каталог `data/uploads`, извлекает текст, формирует `preview` и возвращает расширенные метаданные документа;
- `GET /api/documents` возвращает список загруженных документов из памяти вместе с путем хранения, длиной извлеченного текста и кратким превью;
- `POST /api/ask` принимает вопрос и необязательный `document_id`, но пока не ищет документ и не анализирует его содержимое, а возвращает тестовый ответ.

Что еще не реализовано:

- извлечение текста из `PDF`, `Markdown`, `TXT`;
- связь `document_id` с реальным поиском по документу;
- разбиение текста на чанки;
- векторное хранилище;
- реальный поиск по документу;
- генерация ответа на основе текста документа;
- интеграция с Telegram.

Текущее состояние:

- загруженные файлы теперь сохраняются в `data/uploads`, но список документов по-прежнему хранится только в памяти;
- после перезапуска приложения файлы на диске останутся, но in-memory список документов очистится;
- текст документа уже извлекается, но полноценного индекса для поиска пока нет;
- endpoint `/api/ask` пока не связан с endpoint `/api/upload` на уровне реальной бизнес-логики;
- текущая реализация подходит как учебный и архитектурный старт, но еще не решает основную задачу проекта.

Для запуска текущей версии нужны зависимости:

```bash
pip install fastapi uvicorn pydantic pydantic-settings python-dotenv python-multipart pypdf
```

Примеры запуска и проверки:

```bash
uvicorn app.main:app --reload
```

```bash
curl http://127.0.0.1:8000/api/health
```

```bash
curl -X POST http://127.0.0.1:8000/api/ask \
  -H "Content-Type: application/json" \
  -d '{"question":"What is RAG?"}'
```

```bash
curl -X POST http://127.0.0.1:8000/api/upload \
  -F "file=@example.txt"
```

```bash
curl http://127.0.0.1:8000/api/documents
```

## Текущая структура проекта

```text
AIKnowledgeAssistant/
├── app/
│   ├── api/
│   │   └── routes.py
│   ├── core/
│   │   └── config.py
│   ├── db/
│   │   └── memory_store.py
│   ├── schemas/
│   │   ├── document.py
│   │   └── query.py
│   ├── services/
│   │   ├── document_service.py
│   │   └── rag_service.py
│   └── main.py
├── data/
│   └── uploads/
├── .env
├── .gitignore
└── README.md
```

Структура разделена по ролям:

- `api` - маршруты и HTTP-слой;
- `schemas` - входные и выходные модели данных;
- `services` - бизнес-логика;
- `db` - простое временное хранилище в памяти;
- `core` - конфигурация приложения;
- `data/uploads` - каталог, в который сохраняются загруженные файлы.
